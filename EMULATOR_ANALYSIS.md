## 2. 問題点・ボトルネック

### 2.1 致命的な問題（ゲームが正常動作しない原因）

#### 🔴 P1: Waveチャンネルの周期が8倍間違い
- **ファイル**: `Sound.kt` (周期計算部)
- **現状**: `period = (2048.0 - frequency) * 16.0`
- **正解**: `period = (2048.0 - frequency) * 2.0`
- **影響**: Wave音が本来の3オクターブ上で再生される。多くのゲームの音楽が崩壊する
- **修正コスト**: 低（1行変更）

#### 🔴 P2: Square波のデューティパターンが実機と異なる
- **ファイル**: `Sound.kt` (デューティパターン定義)
- **現状**:
  ```
  Duty 1: 0b00000011  (25%)
  Duty 2: 0b00001111  (50%)
  Duty 3: 0b11111100  (75%)
  ```
- **正解** (Pan Docs準拠):
  ```
  Duty 1: 0b10000001  (25%)
  Duty 2: 0b10000111  (50%)
  Duty 3: 0b01111110  (75%)
  ```
- **影響**: Square波の音質・音色が実機と大きく異なる。特にDuty 1とDuty 2で顕著
- **修正コスト**: 低（定数値の変更）

#### 🔴 P3: DMA転送が即時完了
- **ファイル**: `Ppu.kt`, `SystemBus.kt`
- **現状**: OAM DMA (0xFF46書き込み) で160バイトが即座にコピーされる
- **正解**: 160 CPUサイクルかけて1バイトずつ転送。転送中はHRAM以外アクセス不可
- **影響**: DMAタイミングに依存するゲームが正しく動作しない。多くのゲームはDMA転送中にHRAMで待機ループを回すが、即時完了だとタイミングがずれる
- **修正コスト**: 中

#### 🔴 P4: HALTバグの実装が不完全
- **ファイル**: `Cpu.kt` (HALT処理)
- **現状**: `haltBugActive`フラグは設定されるが、次命令のダブルフェッチが未実装。PCが1バイト進んでしまう
- **正解**: IME=0かつ(IE & IF)≠0のとき、PCを進めずに次のオペコードを2回読む
- **影響**: HALTバグに依存するゲーム（一部の商用タイトル）が暴走する
- **修正コスト**: 中

#### 🔴 P5: HALT中のサイクル進行が不正確
- **ファイル**: `Machine.kt`, `GameBoyCoreImpl.kt`
- **現状**: HALT中は毎回4サイクル固定で返す。さらにHALT最適化で残りサイクルをPPUに一括投入
- **正解**: HALT中は割り込み発生まで各コンポーネントがサイクル単位で同期して進行する
- **影響**: HALT中にPPUだけが高速進行し、Timer/Soundが取り残される。タイマー割り込みのタイミングがずれ、ゲームのテンポが崩れる
- **修正コスト**: 高（アーキテクチャ変更が必要）

### 2.2 高優先度の問題（多くのゲームに影響）

#### 🟡 P6: フレームシーケンサがDIVレジスタと非同期
- **ファイル**: `Sound.kt`
- **現状**: DIVの内部bit 12の立ち下がりエッジで更新（部分的に実装済み）
- **問題**: DIVリセット時のフレームシーケンサへの影響が不完全
- **影響**: エンベロープ・スイープ・長さカウンタのタイミングが微妙にずれ、音楽のテンポ・音色が実機と異なる

#### 🟡 P7: タイマーのエッジ検出未実装
- **ファイル**: `Timer.kt`
- **現状**: TAC変更時にTIMAカウンタをリセットするだけ
- **正解**: DIVの内部16bitカウンタの特定ビット（TAC選択）の1→0遷移でTIMAをインクリメント
- **影響**: タイマー割り込みのタイミングがずれる。タイマー依存のゲーム（テトリスのドロップ速度等）に影響

#### 🟡 P8: PPU Mode 3の固定タイミング
- **ファイル**: `Ppu.kt`
- **現状**: Mode 3 (Pixel Transfer) が固定200サイクル
- **正解**: 172〜289サイクル（スプライト数・SCX mod 8・ウィンドウの有無で変動）
- **影響**: STAT割り込みのタイミングがずれ、ラスタエフェクトを使うゲームで表示が崩れる

#### 🟡 P9: 割り込みサービス中にPPUが更新されない
- **ファイル**: `Machine.kt`
- **現状**: 割り込み処理の20サイクル中、timer.step()とsound.step()のみ呼ばれ、ppu.step()が呼ばれない
- **影響**: 割り込みが頻繁に発生するゲームでPPUタイミングが累積的にずれる

#### 🟡 P10: スプライト描画順序が不正
- **ファイル**: `Ppu.kt`
- **現状**: X座標降順で描画
- **正解**: DMGではX座標が同じ場合、OAMインデックス昇順が優先
- **影響**: 重なり合うスプライトの表示優先度が実機と異なる

### 2.3 中優先度の問題（特定ゲーム・精度に影響）

#### 🟢 P11: ウィンドウ描画の不完全性
- レジスタ(WY/WX)は読み書きできるが、描画ロジックが不完全
- メニュー画面やHUD表示が正しく描画されないゲームがある

#### 🟢 P12: APUの浮動小数点累積誤差
- `soundCycleCounter`がDoubleで無限に増加し、長時間プレイで精度が低下
- 1e9で正規化しているが、正規化時に位相の不連続が発生する可能性

#### 🟢 P13: セーブデータ永続化の未実装
- バッテリバックアップRAMの保存/読み込みが未実装
- ポケモン、ゼルダ等のセーブ機能を持つゲームでデータが失われる

#### 🟢 P14: シリアル通信の即時完了
- 転送開始即座にSC[7]クリア＋割り込み発生
- リンクケーブル通信を行うゲームで問題が発生する可能性

#### 🟢 P15: ステレオ出力がモノラル
- 左右チャンネルの平均を取ってモノラル出力
- ゲームボーイのステレオ効果が失われる

### 2.4 設計上のボトルネック

#### ⚙️ B1: HALT最適化が他コンポーネントを破壊
```
GameBoyCoreImpl.kt (HALT検出部):
  HALT検出 → 残り全サイクルをPPUに一括投入
  → Timer/Soundはその間更新されない
```
これはフレーム実行ループの根本的な問題。HALT中もTimer/Sound/PPUを均等にステップさせる必要がある。

#### ⚙️ B2: 命令単位の同期（T-cycle精度の欠如）
現在は命令実行→結果サイクル数で各コンポーネントを更新する方式。実機はT-cycle (4MHz) 単位で全コンポーネントが同期動作する。命令の途中でPPUモード遷移やタイマーオーバーフローが発生するケースを再現できない。

#### ⚙️ B3: PCスタック検出の副作用
101回の同一PC検出で大量のログを出力。デバッグ時のパフォーマンス低下の原因になる。また、正当なHALT待機を誤検出する可能性がある。
