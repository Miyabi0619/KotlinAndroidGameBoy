# 実機比較レポート：現在の実装 vs Game Boy (DMG) 実機仕様

> 最終更新: 2026-03-20
> 対象コミット: `06a9571` (main)

本ドキュメントは、現在のエミュレータ実装を Pan Docs / TCAGBD / 実機テストROM結果と照合し、
すべての差異を網羅的に列挙したものである。

---

## 凡例

| 記号 | 意味 |
|------|------|
| ✅ | 実機準拠（修正済み or 正しく実装済み） |
| ⚠️ | 部分的に実装されているが不正確 |
| ❌ | 未実装 or 明確に誤っている |
| 🔴 | クリティカル（ゲーム動作に直接影響） |
| 🟡 | 高（特定ゲームで不具合が発生しうる） |
| 🟢 | 中（精度テストで検出されるレベル） |
| 🔵 | 低（エッジケース / CGB専用） |

## 1. CPU (Cpu.kt)

### ✅ 修正済み・正しく実装されている項目

| 項目 | 詳細 |
|------|------|
| HALTバグ | `(IE & IF) != 0` チェック、ダブルフェッチ (`haltBugActive`) 実装済み (L149-158) |
| EI遅延 | `imeEnablePending` で次命令実行後にIME有効化 (L125-128) |
| 割り込み優先度 | VBLANK > LCD_STAT > TIMER > SERIAL > JOYPAD (InterruptController) |
| 割り込みサービス | 20サイクル消費、PCプッシュ+ベクタジャンプ |
| DAA命令 | BCD補正ロジック実装済み |

### ⚠️ 不正確な実装

#### 1.1 🟡 HALT中のサイクル消費が固定4サイクル
- **ファイル**: `Cpu.kt:132-133`
- **現状**: `if (halted || stopped) return Cycles.NOP` — 常に4サイクルを返す
- **実機**: HALT中もDIV/Timer/PPU/APUは正常に動作し続ける。CPUは割り込み発生まで待機するが、その間もサイクルは正しく経過する
- **影響**: HALT→割り込み発生までの時間が実機と異なる。ただし `Machine.stepInstruction()` では各命令後に全コンポーネントを更新しているため、4サイクル刻みで全コンポーネントが進むので実質的な影響は小さい
- **重要度**: 🟡 — HALT中のサイクル精度に依存するゲームで問題が出る可能性

#### 1.2 🟢 STOP命令の挙動が簡略化
- **ファイル**: `Cpu.kt:132`
- **現状**: HALTと同様に4サイクルNOPとして扱う
- **実機**: STOP中はCPU・タイマが完全に停止、ジョイパッド入力で復帰。LCDも停止する
- **影響**: STOPを使用するゲームはDMGではほぼ皆無（CGBのダブルスピード切り替え用途が主）
- **重要度**: 🟢

#### 1.3 🔵 未使用オペコードの処理
- **ファイル**: `Cpu.kt` (executeByOpcode)
- **現状**: `IllegalStateException` をスロー
- **実機**: 0xD3, 0xDB, 0xDD, 0xE3, 0xE4, 0xEB, 0xEC, 0xED, 0xF4, 0xFC, 0xFD — ロックアップまたはNOP相当
- **重要度**: 🔵 — 正規のゲームでは使用されない

## 2. PPU (Ppu.kt)

### ✅ 修正済み・正しく実装されている項目

| 項目 | 詳細 |
|------|------|
| モード遷移タイミング | Mode 2 (80T) → Mode 3 (可変) → Mode 0 (残り) → Mode 1 (VBlank) |
| Mode 3 可変サイクル | SCX mod 8 + スプライト数 × 6 + ウィンドウ有無で計算 (L454-494) |
| STAT割り込み | エッジ検出方式で実装、HBlank/VBlank/OAM/LYC=LY各ソース対応 (L401-442) |
| VRAM/OAMアクセス制限 | Mode 3中VRAM不可、Mode 2/3中OAM不可 (L100-110) |
| LYC比較 | LYCラップアラウンド対応 (`lyc % 154`) |
| LCD OFF挙動 | LY=0、Mode=HBLANK、画面白 (L519以降) |
| 背景描画 | BGマップ0/1、スクロール、タイルデータモード（符号付き/なし） |
| スプライト表示制限 | 1スキャンライン最大10個 |
| 8x16スプライト | タイルインデックス下位ビットマスク対応 (L825-830) |
| スプライトBG優先度 | `bgColorIds` を使ったカラーID判定 (L853-857) |
| ウィンドウ描画 | WX-7/WY座標、BGの上に描画、カラーID更新 (L641-719) |

### ⚠️ 不正確な実装

#### 2.1 🔴 スプライト優先度：X座標によるソートが未実装
- **ファイル**: `Ppu.kt:805-808`
- **現状**: `spritesOnLine.reversed()` — OAMインデックスの降順で描画（低インデックスが上に来る）
- **実機 (DMG)**: 同一ピクセルに複数スプライトが重なる場合、**X座標が小さいスプライトが優先**。X座標が同じ場合のみOAMインデックスで決定
- **影響**: スプライトが重なるシーン（キャラクター同士の重なり、弾幕等）で描画優先度が実機と異なる
- **修正方法**: OAM Search後に `sortedBy { oam[it * 4 + 1].toInt() }` でX座標昇順ソート（安定ソートでOAMインデックス順を保持）し、逆順で描画
- **重要度**: 🔴 — 多くのゲームでスプライト重なりの表示が異なる

#### 2.2 🟡 ウィンドウ内部ラインカウンタの不正確さ
- **ファイル**: `Ppu.kt:676-678`
- **現状**: `windowY = y - windowStartY` — 画面Y座標からWYを引いた値を使用（VBlank時に一括描画）
- **実機**: ウィンドウには専用の内部ラインカウンタがあり、以下の挙動を持つ：
  - ウィンドウが実際に表示されたスキャンラインでのみインクリメント
  - フレーム途中でウィンドウを無効化→再有効化しても、カウンタは途中から継続
  - WY変更で一度ウィンドウ表示が始まると、WYを変更してもカウンタはリセットされない
- **影響**: ウィンドウの表示/非表示をフレーム中に切り替えるゲーム（スプリットスクリーン、ステータスバー等）で描画がずれる
- **修正方法**: `step()` 内のスキャンラインループでウィンドウ内部カウンタを管理し、`captureFrameInternal()` ではそのカウンタを参照
- **重要度**: 🟡

#### 2.3 🟡 VBlank時一括描画 vs スキャンラインごと描画
- **ファイル**: `Ppu.kt:518` (`captureFrameInternal`)
- **現状**: VBlank開始時（LY=144）にVRAM/OAMの現在状態からフレーム全体を一括描画
- **実機**: 各スキャンラインはMode 3中にリアルタイムで描画される。ゲームがHBlank中にVRAM/OAM/レジスタを書き換えると、次のスキャンラインから反映される
- **影響**: HBlankタイミングでスクロールレジスタやパレットを変更する「ラスターエフェクト」が正しく描画されない場合がある。SCX/SCYはスキャンラインごとにキャプチャしているが、BGP/OBP/LCDCの変更は反映されない
- **修正方法**: `step()` 内でMode 3開始時にスキャンライン単位で描画し、その時点のレジスタ値を使用
- **重要度**: 🟡 — ラスターエフェクト使用ゲーム（一部のシューティング、デモ等）に影響

#### 2.4 🟡 ウィンドウWX < 7の処理
- **ファイル**: `Ppu.kt:653-654`
- **現状**: `if (wxInt < 7) return` — WX < 7の場合ウィンドウ非表示
- **実機**: WX=0-6でもウィンドウは表示されるが、バグのある挙動が発生する（WX=0でウィンドウが1ピクセル右にずれる等）
- **影響**: WX < 7を使用するゲーム（少数だが存在）でウィンドウが表示されない
- **重要度**: 🟡

#### 2.5 🟢 ピクセルFIFO未実装
- **現状**: タイルマップから直接ピクセル値を計算してフレームバッファに書き込む
- **実機**: 背景FIFOとスプライトFIFOの2本のFIFOを持ち、ピクセルごとにマージして出力
- **影響**: サイクル精度のテストROM（例: `dmg-acid2`）で不合格。ただし通常のゲームでは視覚的な差異はほぼない
- **重要度**: 🟢

#### 2.6 🟢 Mode 3のスプライトフェッチペナルティが近似値
- **ファイル**: `Ppu.kt:482-483`
- **現状**: `spriteCount * 6` で固定
- **実機**: スプライトのX座標に基づいて6-11サイクルの可変ペナルティ（SCXとの位置関係に依存）
- **重要度**: 🟢

## 4. APU / サウンド (Sound.kt)

### ✅ 修正済み・正しく実装されている項目

| 項目 | 詳細 |
|------|------|
| フレームシーケンサDIV同期 | DIV内部カウンタのbit 12立ち下がりエッジで512Hz更新 (L657-664) |
| デューティパターン | Pan Docs準拠: 12.5%=0b00000001, 25%=0b10000001, 50%=0b10000111, 75%=0b01111110 (L1011-1026) |
| Square周期計算 | `(2048 - freq) * 4.0` サウンドサイクル（全8ステップ周期として正しい） |
| Wave周期計算 | `(2048 - freq) * 8.0` サウンドサイクル（全32サンプル周期として正しい） (L1338) |
| 長さカウンタ | 256Hz更新、フレームシーケンサstep 0,2,4,6で実行 |
| エンベロープ | 64Hz更新、step 7で実行。0-15範囲制限。DAC無効チェック |
| NR52書き込み | bit7=0で全チャンネル無効化+レジスタクリア（Wave RAMは保持） |
| DAC無効化 | NRx2上位5ビット全0でチャンネル無効 (L383-385, L400-403) |
| Noise LFSR | 15bit/7bitモード、XOR計算、半サウンドサイクル単位の精密なタイミング |
| DCブロッキングフィルタ | HPフィルタで実機のアナログコンデンサをモデル化 (L930-938) |
| ステレオ出力 | L/R独立ミキシング、NR51チャンネル選択、NR50マスターボリューム |
| Wave RAMスナップショット | トリガー時にWave RAMをコピーして使用 |

### ⚠️ 不正確な実装

#### 4.1 🔴 Sweepのダブルオーバーフローチェック欠如
- **ファイル**: `Sound.kt:1472-1497`
- **現状**: 周波数更新後にオーバーフローチェックを1回のみ実行。更新が成功した場合そのまま終了
- **実機**: 周波数更新が成功した後、**再度オーバーフロー計算を行い（実際には書き込まない）**、この2回目のチェックでも2047を超えたらチャンネルを無効化する
- **影響**: ポケモン赤/緑等でスイープを使った音が正しく停止しないケースがある
- **修正方法**:
  ```kotlin
  // 周波数更新成功後に2回目のオーバーフローチェック
  val secondCheck = if (sweepNegate) {
      newFrequency - (newFrequency shr currentSweepShift)
  } else {
      newFrequency + (newFrequency shr currentSweepShift)
  }
  if (secondCheck > 2047) {
      square1State.enabled = false
      sweepEnabled = false
  }
  ```
- **重要度**: 🔴

#### 4.2 🟡 Sweep change=0 のスキップ
- **ファイル**: `Sound.kt:1475-1478`
- **現状**: `if (change == 0) return` — shift結果が0の場合更新をスキップ
- **実機**: change=0でもshadowFrequencyの更新とオーバーフローチェックは実行される
- **影響**: sweepShift=0でnegate=trueの場合のチャンネル無効化挙動が実機と異なる
- **重要度**: 🟡

#### 4.3 🟡 エンベロープ「ゾンビモード」未実装
- **ファイル**: `Sound.kt:377-385`
- **現状**: NR12/NR22/NR42への書き込み時に即座にエンベロープ状態を更新
- **実機**: チャンネルがアクティブな状態でエンベロープレジスタに書き込むと、「ゾンビモード」と呼ばれる特殊な挙動が発生：
  - ボリュームの即時変更（+1/-1）
  - 方向ビット反転時のボリューム反転 (15 - volume)
  - パルス幅変更に似た離散的な音色変化
- **影響**: ゾンビモードを利用した音声エフェクトが正しく再生されない
- **重要度**: 🟡

#### 4.4 🟡 長さカウンタの追加クロッキング
- **ファイル**: `Sound.kt:388-391`
- **現状**: NRx4のbit 6（Length Enable）変更時に追加クロッキングなし
- **実機**: フレームシーケンサが特定のステップ（奇数ステップ：長さカウンタが更新されないステップ）にあるとき、Length Enableを0→1に変更すると**追加の長さカウンタデクリメント**が発生する
- **影響**: 長さカウンタの精度テストROM（blargg dmg_sound 03-trigger）で不合格
- **重要度**: 🟡

#### 4.5 🟡 Wave チャンネルトリガー時の波形RAM破壊（DMG固有）
- **ファイル**: `Sound.kt:541-558`
- **現状**: トリガー時にWave RAMを正常にスナップショット
- **実機 (DMG)**: Waveチャンネルが波形RAMを読み取り中（チャンネルが再生中）にトリガーすると、波形RAMの最初の4バイトが読み取り中の位置から**破壊的にコピー**される
- **影響**: Waveチャンネルの再トリガーを高速に行うゲーム（PCMストリーミング的手法）で音が実機と異なる
- **重要度**: 🟡

#### 4.6 🟡 Wave チャンネルの高周波数カットオフ
- **ファイル**: `Sound.kt:1327`
- **現状**: `frequency >= 2044` で無音化
- **実機**: 全周波数値で音声が出力される（超音波域含む）。高周波数でのカットオフはアナログ回路の特性であり、デジタル段階では行わない
- **影響**: freq=2044-2047を使うゲーム（スキャンラインPCM手法の一部）で無音になる可能性
- **重要度**: 🟡

#### 4.7 🟢 NR52読み取り時のチャンネルステータス
- **ファイル**: `Sound.kt:341-363`
- **現状**: NR52のbit 0-3にチャンネルのenabled状態を反映
- **実機**: チャンネルのDAC状態とenabled状態の両方が関係する（DACがオフならチャンネルは無効として報告）
- **重要度**: 🟢

#### 4.8 🟢 サンプル生成の浮動小数点累積誤差
- **ファイル**: `Sound.kt` (phaseAccumulator)
- **現状**: `Double` 型の位相アキュムレータ。長時間再生で累積誤差の可能性
- **実機**: 整数カウンタベース。誤差なし
- **影響**: 数時間の連続再生で微小なピッチドリフトの可能性
- **重要度**: 🟢

## 5. メモリバス (SystemBus.kt)

### ✅ 修正済み・正しく実装されている項目

| 項目 | 詳細 |
|------|------|
| メモリマップ | ROM/VRAM/WRAM/OAM/IO/HRAM/IE 正しくマッピング |
| Echo RAM | C000-DDFFをE000-FDFFにミラーリング |
| VRAM/OAMアクセス制限 | PPUモードに基づくアクセスブロック |
| OAM DMA転送 | 160バイト/640 T-cycle、HRAM以外アクセス制限 |
| MBC1/MBC3/MBC5 | カートリッジタイプ別に対応 |

### ⚠️ 不正確な実装

#### 5.1 🟢 DMAバス競合の簡略化
- **現状**: DMA中のメモリアクセスは単純にブロック（0xFFを返す）
- **実機**: DMA中もCPUはバスを使おうとし、DMAソースアドレスの値が読み取られるケースがある
- **重要度**: 🟢

#### 5.2 🟡 シリアル通信の即時完了
- **ファイル**: `SystemBus.kt`（シリアル転送）
- **現状**: SC[7]=1で即座にSB=0xFF、割り込み発生
- **実機**: 内部クロック選択時は8192サイクル（512Hz × 8bit）かけて転送
- **影響**: シリアル通信タイミングに依存するゲーム（テトリスのリンク対戦等）で問題
- **重要度**: 🟡

#### 5.3 🔵 ブートROM非対応
- **現状**: PC=0x0100から直接開始、レジスタを手動で初期化
- **実機**: 0x0000-0x00FFにブートROM、起動アニメーション後にFF50レジスタで無効化
- **影響**: 起動画面が表示されない（ゲーム動作には影響なし）
- **重要度**: 🔵

## ポケモンピカチュウ 鳴き声ノイズについて

ポケモンピカチュウの鳴き声は、Wave チャンネルを高速にトリガーして PCM ストリーミングを行う手法で実現されている。以下の問題が複合的に影響している可能性がある：

1. **Wave 高周波数カットオフ** (4.6): freq≥2044で無音化すると、鳴き声の切り替え時に不要なギャップが発生
2. **Wave トリガー時の波形RAM破壊** (4.5): DMG固有の挙動が未実装のため、高速トリガー時の波形データが実機と異なる
3. **Sweep ダブルオーバーフローチェック** (4.1): Square 1 の鳴き声パートが正しく停止しない可能性

これらの修正を Phase 1/2 の優先度で進めることを推奨する。
